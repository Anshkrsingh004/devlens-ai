# DevLens AI — Database Design

> PostgreSQL (Supabase free tier) · Prisma 7 ORM
> Companion to `IMPLEMENTATION_ROADMAP.md`

---

## 1. Design Principles

1. **Better Auth owns its tables.** `user`, `session`, `account`, and `verification` are generated by the Better Auth CLI and never hand-edited. Our models reference `user.id` but never modify that table's shape.
2. **Normalize relationships, not payloads.** Foreign keys and enums are relational. The AI report is a write-once, read-whole document and stays as JSON.
3. **Index for the queries that actually run.** Every index below maps to a specific screen. Speculative indexes cost write throughput and are omitted.
4. **Cascade deletes from `user`.** Deleting an account removes all its data — GDPR-friendly and prevents orphans.
5. **Every user-scoped query filters by `userId` in the `where` clause.** Never fetch-then-compare.

## 3. Tables

### 3.1 `Review`

The core entity. One row per review request, created **before** the AI call so nothing is ever silently lost.

| Column             | Type            | Null | Default      | Purpose                                        |
| ------------------ | --------------- | ---- | ------------ | ---------------------------------------------- |
| `id`               | `String` (cuid) | ✗    | `cuid()`     | Primary key                                    |
| `userId`           | `String`        | ✗    | —            | FK → `user.id`, cascade delete                 |
| `title`            | `String(120)`   | ✗    | —            | AI-suggested, user-editable                    |
| `language`         | `Language`      | ✗    | —            | Enum, drives prompt + highlighting             |
| `sourceCode`       | `Text`          | ✗    | —            | Submitted source, **capped at 8,000 chars**    |
| `sourceType`       | `SourceType`    | ✗    | `PASTE`      | Provenance: paste or upload                    |
| `status`           | `ReviewStatus`  | ✗    | `PENDING`    | Lifecycle state                                |
| `result`           | `Json`          | ✓    | `null`       | Full validated report. Null until complete     |
| `overallScore`     | `Int`           | ✓    | `null`       | **Denormalized** from `result` for sort/filter |
| `isFavorite`       | `Boolean`       | ✗    | `false`      | User bookmark                                  |
| `errorCode`        | `String(64)`    | ✓    | `null`       | Taxonomy code when `FAILED`                    |
| `model`            | `String(64)`    | ✓    | `null`       | Model used — enables A/B comparison later      |
| `promptTokens`     | `Int`           | ✓    | `null`       | Observability                                  |
| `completionTokens` | `Int`           | ✓    | `null`       | Observability                                  |
| `latencyMs`        | `Int`           | ✓    | `null`       | Observability                                  |
| `createdAt`        | `DateTime`      | ✗    | `now()`      | Sort key                                       |
| `updatedAt`        | `DateTime`      | ✗    | `@updatedAt` | Audit                                          |

**Constraints**

- `sourceCode` length enforced at the application layer via a shared Zod schema (`MAX_SOURCE_CHARS = 8000`). Postgres `Text` has no practical limit, so the guarantee lives in the one place both client and server read.
- `overallScore` is expected in `0…100`; validated by Zod on write.

**Indexes**

| Index                               | Serves                                          |
| ----------------------------------- | ----------------------------------------------- |
| `@@index([userId, createdAt DESC])` | Dashboard list — the single most frequent query |
| `@@index([userId, isFavorite])`     | Favorites filter                                |
| `@@index([userId, status])`         | Surfacing `FAILED` reviews for retry            |
| GIN `pg_trgm` on `title`            | `ILIKE '%q%'` search (added in M5)              |

> **Why every index leads with `userId`.** Every query in the application is user-scoped. A composite index starting with `userId` serves both the filter and the sort in one index scan. An index on `createdAt` alone would be useless here.

---

#### Design decision: `result` as `Json`

**Why not normalize the 13 sections into tables?**

The report is written exactly once and always read whole. It is never partially queried, never independently updated, and never joined against. Normalizing it into `ReviewBug`, `ReviewSecurityIssue`, `ReviewSmell`, and so on would produce:

- 6–8 additional tables
- A multi-join query on every review page load
- A migration every time the report shape evolves

…for **zero** query benefit.

**Is type safety lost?** No. The blob is `ReviewResultSchema.parse()`-ed on write _and_ on read. Prisma types it as `Json`, and the repository casts through Zod, so consumers receive a fully typed `ReviewResult`. The guarantee is stronger than a foreign key because it validates shape _and_ content.

**When would this be wrong?** If we ever needed to query _across_ reviews by finding — "show every SQL-injection issue I've ever had". That is enhancement F5-adjacent. At that point, add a narrow `ReviewFinding` projection table populated on write, keeping `result` as the source of truth. The migration is additive and non-breaking, which is exactly why this decision is safe today.

#### Design decision: `overallScore` denormalized

This is a deliberate, documented exception to "never duplicate data".

`overallScore` also lives inside `result.overallScore`. It is duplicated into a real column because it is the **one** field we sort and filter by. Sorting inside a JSON column requires an expression index and defeats the query planner in Postgres; a plain `Int` column is trivially indexable.

Duplication is safe because both values are written in the same transaction from the same validated object — they cannot drift.

### 3.2 `DailyUsage`

Per-user, per-day rate limiting. **This is the Redis replacement.**

| Column        | Type            | Null | Default      | Purpose                         |
| ------------- | --------------- | ---- | ------------ | ------------------------------- |
| `id`          | `String` (cuid) | ✗    | `cuid()`     | Primary key                     |
| `userId`      | `String`        | ✗    | —            | FK → `user.id`, cascade         |
| `date`        | `Date`          | ✗    | —            | Day bucket (date only, no time) |
| `reviewCount` | `Int`           | ✗    | `0`          | Reviews attempted today         |
| `tokensUsed`  | `Int`           | ✗    | `0`          | Tokens consumed today           |
| `createdAt`   | `DateTime`      | ✗    | `now()`      | —                               |
| `updatedAt`   | `DateTime`      | ✗    | `@updatedAt` | —                               |

**Constraints**

- `@@unique([userId, date])` — makes the row atomically upsertable and prevents double-counting under concurrency.

**Indexes** — the unique constraint is the index. No others needed; this table is only ever read by exact `(userId, date)` lookup.

> **Why a table instead of Redis?** Redis is explicitly banned by the ₹0 constraint, and it is genuinely unnecessary here. An atomic `upsert` with `increment` on a unique `(userId, date)` key is a correct, durable, free counter. Postgres handles the concurrency; we get persistence and history for free — and the same rows power the settings-page usage meter without a second data source.
>
> **Why daily rather than a sliding window?** Groq's free tier resets daily (1,000 RPD / 200,000 TPD). Matching our bucket to the provider's bucket makes the quota trivially explainable to the user: "8 of 10 reviews used today."

**Retention.** Rows older than 90 days can be pruned; there is no need to keep them and the 500 MB cap deserves respect. Not implemented in the MVP — at ~60 bytes per row, this is a non-issue until well past portfolio scale.

### 3.3 `UserPreference`

One row per user, created automatically at sign-up.

| Column                    | Type            | Null | Default      | Purpose                                    |
| ------------------------- | --------------- | ---- | ------------ | ------------------------------------------ |
| `id`                      | `String` (cuid) | ✗    | `cuid()`     | Primary key                                |
| `userId`                  | `String`        | ✗    | —            | FK → `user.id`, **unique**, cascade        |
| `theme`                   | `Theme`         | ✗    | `SYSTEM`     | UI theme                                   |
| `defaultLanguage`         | `Language`      | ✗    | `TYPESCRIPT` | Pre-selected in the workspace              |
| `reviewDepth`             | `ReviewDepth`   | ✗    | `BALANCED`   | Alters prompt verbosity **and token cost** |
| `includeRefactor`         | `Boolean`       | ✗    | `true`       | Token-saving switch                        |
| `createdAt` / `updatedAt` | `DateTime`      | ✗    | —            | —                                          |

**Constraints** — `userId` is `@unique`, enforcing 1:1.

> **Why a separate table instead of extending `user`?** Better Auth generates and owns the `user` table. Adding columns to it couples our schema to their code generation — a future CLI run or version bump could conflict. A 1:1 side table isolates that blast radius completely, at the cost of one join we rarely need (preferences load once per session).
>
> **Why `reviewDepth` and `includeRefactor` matter.** These are not cosmetic. They are the user-facing controls over the 8,000 TPM budget: turning off refactored code reclaims roughly 1,500 output tokens, which is the difference between a review succeeding and hitting a 429 on a large file.

### 3.4 Better Auth tables (generated — reference only)

Generated by `npx @better-auth/cli generate`. **Never hand-edited.**

| Table          | Purpose                                                       |
| -------------- | ------------------------------------------------------------- |
| `user`         | Identity: id, name, email, emailVerified, image, timestamps   |
| `session`      | Active sessions: token, expiresAt, ipAddress, userAgent       |
| `account`      | Credential and OAuth provider links (Google), password hashes |
| `verification` | Email verification and password reset tokens                  |

> If Better Auth's schema changes on upgrade, re-run the CLI and review the generated migration. Because none of our columns live in these tables, such a migration can never conflict with application data.

## 4. Enums

| Enum           | Values                                                  | Notes                                                          |
| -------------- | ------------------------------------------------------- | -------------------------------------------------------------- |
| `Language`     | `CPP` · `PYTHON` · `JAVA` · `JAVASCRIPT` · `TYPESCRIPT` | Adding a value is an additive migration                        |
| `SourceType`   | `PASTE` · `UPLOAD`                                      | Extensible to `GITHUB_PR` for enhancement F2                   |
| `ReviewStatus` | `PENDING` · `COMPLETED` · `FAILED`                      | `PENDING` exists because the row is written before the AI call |
| `Theme`        | `LIGHT` · `DARK` · `SYSTEM`                             | —                                                              |
| `ReviewDepth`  | `CONCISE` · `BALANCED` · `THOROUGH`                     | Directly controls token spend                                  |

> **Why database enums rather than strings?** Postgres enums are validated at the database level, self-documenting in the schema, and generate TypeScript union types through Prisma. A typo becomes a compile error instead of a row nobody notices for three weeks.
>
> **The trade-off:** adding an enum value requires a migration. Since these sets change rarely and deliberately, that friction is a feature — it forces the corresponding prompt and UI updates to ship together.

## 5. Query Patterns

Every query the application actually issues, and the index that serves it.

| #   | Query                                           | Screen                 | Index used                          |
| --- | ----------------------------------------------- | ---------------------- | ----------------------------------- |
| Q1  | Reviews by user, newest first, cursor-paginated | Dashboard              | `[userId, createdAt DESC]`          |
| Q2  | Same, filtered to favorites                     | Dashboard filter       | `[userId, isFavorite]`              |
| Q3  | Same, filtered by language                      | Dashboard filter       | `[userId, createdAt DESC]` + filter |
| Q4  | Title search `ILIKE '%q%'`                      | Search                 | GIN `pg_trgm` on `title`            |
| Q5  | Single review by `(id, userId)`                 | Review page            | Primary key + `userId` filter       |
| Q6  | Usage for `(userId, today)`                     | Quota check            | `UNIQUE(userId, date)`              |
| Q7  | Upsert + increment usage                        | After review           | `UNIQUE(userId, date)`              |
| Q8  | Preferences by `userId`                         | Settings, prompt build | `UNIQUE(userId)`                    |
| Q9  | `FAILED` reviews for a user                     | Retry surfacing        | `[userId, status]`                  |

**Cursor pagination, not offset.** `OFFSET` grows linearly slower and — more importantly — produces duplicate or skipped rows when items are inserted between page fetches. Cursor pagination on `(createdAt, id)` is stable under concurrent writes and index-friendly.

**Security note on Q5.** The lookup is always `where: { id, userId }`, never `findUnique({ id })` followed by an ownership comparison in application code. A missing row and an unowned row are indistinguishable to the caller, which returns `404` and leaks nothing about whether the ID exists. Fetch-then-compare is how IDOR vulnerabilities are written.

## 6. Data Volume & Free-Tier Capacity

Supabase free tier provides **500 MB**.

| Table                                 | Bytes/row (est.) | 10,000 reviews |
| ------------------------------------- | ---------------: | -------------: |
| `Review` (8 KB source + ~6 KB result) |           ~14 KB |        ~140 MB |
| `DailyUsage`                          |            ~60 B |        ~0.2 MB |
| `UserPreference`                      |           ~100 B |     negligible |
| Better Auth tables                    |      ~500 B/user |     negligible |

**Headroom: roughly 30,000 reviews before approaching the cap.** Given the Groq free tier ceiling of ~30 reviews/day account-wide, reaching that would take multiple years. **Storage is not a binding constraint; the AI rate limit is.**

If it ever became one, in order of preference: prune `sourceCode` on reviews older than N days (keeping the result), then archive `FAILED` rows, then compress `result`.

## 7. Migration Strategy

| Environment               | Command                 | Notes                    |
| ------------------------- | ----------------------- | ------------------------ |
| Local development         | `prisma migrate dev`    | Uses `DIRECT_URL`        |
| Production (Vercel build) | `prisma migrate deploy` | Runs before `next build` |
| Client generation         | `prisma generate`       | Runs on `postinstall`    |

**The two-URL requirement is mandatory, not optional:**

| Variable       | Port | Used by                          | Why                                                                 |
| -------------- | ---- | -------------------------------- | ------------------------------------------------------------------- |
| `DATABASE_URL` | 6543 | Prisma Client at runtime         | Supavisor pooling — serverless exhausts direct connections          |
| `DIRECT_URL`   | 5432 | Prisma CLI (migrate, introspect) | Migrations require a session-mode connection; pgbouncer breaks them |

> **Failure mode if reversed or omitted:** migrations hang indefinitely, or production throws `prepared statement "s0" already exists` under concurrency. Both are expensive to diagnose _after_ the fact, which is why M1's acceptance criteria require verifying both URLs against the deployed environment — not just locally.

**Migration rules**

1. Migrations are committed to git and never edited after merge.
2. Additive changes only when possible: add nullable, backfill, then enforce.
3. Never `prisma db push` against production — it bypasses migration history.
4. Every migration must apply cleanly to an empty database (tested in CI).

## 8. Future Scalability

None of these require redesigning existing tables. Each is additive — the test of a sound schema.

| #   | Change                      | Schema impact                                                       | Trigger                        |
| --- | --------------------------- | ------------------------------------------------------------------- | ------------------------------ |
| S1  | GitHub PR review            | Add `GITHUB_PR` to `SourceType`; add nullable `sourceUrl`           | Enhancement F2                 |
| S2  | Public shareable reviews    | Add `shareSlug` (unique, nullable) + `isPublic` to `Review`         | F6                             |
| S3  | Cross-review finding search | New `ReviewFinding` projection table, populated on write            | When users request it          |
| S4  | Team workspaces             | New `Organization` + `Membership`; add nullable `orgId` to `Review` | F8                             |
| S5  | Billing                     | New `Subscription` table; add `plan` to `UserPreference`            | F9                             |
| S6  | Prompt A/B testing          | `model` and token columns already exist — **no migration needed**   | Anytime                        |
| S7  | Multi-file reviews          | New `ReviewFile` table; `Review` becomes the parent                 | F4                             |
| S8  | Soft delete                 | Add `deletedAt`; filter in the repository                           | If undo must outlive the toast |

**The one change that would require real thought** is S7 (multi-file). Today `sourceCode` lives on `Review`, assuming one file per review. Moving to `ReviewFile` means migrating existing rows into child records. It remains mechanical — a data migration, not a redesign — but it is the only item on this list that touches existing data rather than merely adding to it.

**Scaling beyond the free tier**, in the order the constraints would actually bind:

1. **Groq rate limits bind first** — long before anything else. Mitigated by a paid tier or a self-hosted model (F10).
2. **Supabase 500 MB** — years away at the achievable review rate.
3. **Vercel Hobby** — the commercial-use restriction binds before any technical limit does.

Postgres itself would comfortably serve this schema into the millions of rows with the indexes above. **The database is the least likely component to become the bottleneck.**
